Java API Compliance Checker

Java API Compliance Checker (Java ACC) is a tool for checking backward binary and source compatibility of a Java library API. The tool checks classes declarations of old and new versions and analyzes changes that may break compatibility: removed methods, removed class fields, added abstract methods, etc. Binary incompatibility may result in crashing or incorrect behavior of existing clients built with an old version of a library when they are running with a new one. Source incompatibility may result in recompilation errors with a new library version. The tool is intended for library developers and operating system maintainers who are interested in ensuring backward compatibility, i.e. allow old clients to run or to be recompiled with newer library versions.

Supported Platforms

GNU/Linux, FreeBSD, Mac OS X, MS Windows (Xp, Vista, 7).

System Requirements

  • Linux, FreeBSD, Mac OS X
    • JDK (javap, javac)
    • Perl (5.8-5.14)
  • MS Windows
    • JDK (javap, javac)
    • Active Perl (5.8-5.14)

Detectable Compatibility Problems

The tool searches for the following list of changes in the API that may break binary/source compatibility:

  • Removed Elements (methods, classes, interfaces)
  • Problems with Data Types
    • Classes
      • removed/changed methods
      • removed/changed fields
      • added/removed/changed abstract methods
      • added/removed super-interfaces
      • added/removed super-classes
    • Interfaces
      • added/removed/changed abstract methods
      • removed/changed fields
      • added/removed/changed super-interfaces
  • Problems with Methods
    • changed attributes
    • added/removed exceptions
    • moving a method up class hierarchy
    • overridden methods
  • Problems with Implementation
    • changes in decompiled binary code

You can see detailed problem descriptions in the HTML report (see example) generated by the tool.

Limitations

The tool cannot check compatibility problems related to changes in code annotations.

Usage

For using the tool, you should provide the XML descriptors for two library versions: v1.xml and v2.xml files. Library descriptor is a simple XML-file that specifies version number, paths to Java ARchives and other optional information. An example of the descriptor is the following (1.0.xml): 

 <version>
     1.0
 </version>
 
 <archives>
     /path1/to/JAR(s)/
     /path2/to/JAR(s)/
       ...
 </archives>

Compare Libraries

Command to compare two versions of a library:
   japi-compliance-checker -l <library_name> -old <v1.xml> -new <v2.xml> 

Short command to compare two single JARs:
   japi-compliance-checker -l <library_name> -old <v1.jar> -new <v2.jar> 

The compatibility report will be generated to:
   compat_reports/<library_name>/<v1>_to_<v2>/compat_report.html 

Check Clients Portability

Command to check a client portability to a newer version of a Java library API:
   japi-compliance-checker -l <library_name> -old <v1.xml> -new <v2.xml> -client <client.jar> 

Dump Library API to TXT file

Command to dump a Java library API to gzipped TXT format file:
   japi-compliance-checker -l <library_name> -dump <some_version.xml> 

The API dump will be generated to:
   api_dumps/<library_name>/<library_name>_<some_version>.api.tar.gz

Examples

Check the JDO library API versions for compatibility:
   japi-compliance-checker -l jdo -old jdo-api-2.0.jar -new jdo-api-3.0.jar 
The report will be generated to:
   compat_reports/jdo/2.0_to_3.0/compat_report.html 

Dump library API:
   japi-compliance-checker -l jdo -dump jdo-api-2.0.jar 
The API dump will be generated to:
   api_dumps/jdo/jdo_2.0.api.tar.gz 

Check client (Teller.jar) portability between JDO versions:
   japi-compliance-checker -l jdo -old jdo-api-2.0.jar -new jdo-api-3.0.jar -client Teller.jar 

Report Format

The HTML-format compatibility report consists of:

  • Summary - Verdict on compatibility. Number of archives, classes and methods checked by the tool.
  • Problem Summary - Classification of compatibility problems.
  • Added Methods - List of added methods.
  • Removed Methods - List of removed methods.
  • Problems with Data Types - List of compatibility problems caused by changes in data types (divided by the level of risk: High, Medium, Low). List of affected methods.
  • Problems with Methods - List of compatibility problems caused by changes in method parameters and attributes (divided by the level of risk).
  • Problems with Implementation - List of changes in decompiled binary code. Use -check-implementation option to enable this section.

Examples:

  • JDO: 2.2 to 3.0 compatibility report

Verdict on Compatibility

If the tool detected problems with high or medium level of severity or at least one removed symbol then the compatibility verdict is incompatible (otherwise compatible). Low-severity problems can be considered as warnings and don't affect the compatibility verdict unless the -strict option is specified.
